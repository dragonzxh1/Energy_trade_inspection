<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form → /api/trade body | `sellerDomain` is user-provided text input — untrusted string crosses into server-side processing here |
| route.ts → runTradeCheck() → checkDomain() | `sellerDomain` is passed through two layers before reaching the RDAP HTTP call |
| checkDomain() → RDAP servers | `extractDomain()` must be the sole gate before any network call is made with user input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Tampering / SSRF | `sellerDomain` form input → `checkDomain()` | mitigate | `extractDomain()` in `domain-check.ts` parses the input using URL parsing and regex — it returns `null` for non-domain strings, invalid URLs, and IP addresses. `checkDomain()` constructs RDAP URLs as `${RDAP_BASE[tld]}/domain/${domain}` — the domain component is a TLD-validated string, not a free-form URL. Arbitrary SSRF via `sellerDomain` is not possible. Same path already in production via `screening-service.ts`. |
| T-06-05 | Injection | `sellerDomain` → `extractDomain()` output → RDAP URL | mitigate | `extractDomain()` output is a domain label (e.g. `seller.com`) with no URL-scheme, port, or path components. The RDAP base URL is a compile-time constant from `RDAP_BASE` (keyed by TLD). Injection into the URL structure is not possible. |
| T-06-06 | Information Disclosure | RDAP response → `DomainCheckResult.evidence[]` | accept | Evidence strings are generated server-side from RDAP metadata (domain age, privacy status). They are shown to the authenticated compliance officer in TradeClient.tsx FlagCard — the same user who submitted the check. No new cross-user disclosure. |
| T-06-07 | Denial of Service | RDAP call volume via trade check | accept | `checkDomain()` caches results in `domain_whois_cache` (48h TTL). Repeated trade checks for the same domain hit the cache, not the RDAP server. Rate abuse via the trade endpoint already requires authentication + paid plan (enforced in route.ts). |
</threat_model>
