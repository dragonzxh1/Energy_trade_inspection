### Integration Gaps

**GAP-1 — Domain flags unreachable via `/api/trade`** (Functional — Affects DATASRC-05, DATASRC-06, DECISION-03)

`sellerDomainCheck` is defined in `TradeRuleInput` and populated in `screening-service.ts` (document upload flow) but absent from the `runTradeRules()` invocation in `trade-service.ts` (lines 321–348). Consequence: `DOMAIN_WHOIS_RISK` and `DOMAIN_SPOOFING_RISK` flags will never appear in trade check verdicts via `/api/trade`.

Fix: Call `checkDomain(sellerDomain)` and `checkEmailDomain(sellerDomain)` in `runTradeCheck()` and pass results as `sellerDomainCheck` to `runTradeRules()`.

**GAP-2 — Circuit breaker degradation invisible in trade checks** (Functional — Affects ARCH-02)

`trade-service.ts` calls `checkSanctions()` with `.catch(() => ({ listed: false, sources: [] }))` — the `.status` field from the result is never read. When OpenSanctions API is degraded, a trade check returns a verdict with no indication that the sanction check was bypassed. A compliance officer receives potentially misleading confidence.

Fix: Check `sellerSanction.status === 'degraded'` and `vesselSanction.status === 'degraded'` in `trade-service.ts`; add `sanctionDegraded?: boolean` to `TradeCheckResult`.

**GAP-3 — `warninglists` SyncSource untargeted via admin API** (Operational — Affects DATASRC-01 through DATASRC-04)

Calling `POST /api/admin/sync { "source": "warninglists" }` maps to `runSync('all')` and triggers all data source syncs. Warning lists ARE always synced as part of full sync. Isolated targeted sync is unavailable via the HTTP API.

Fix: Add `'warninglists'` and `'legitdomains'` to the explicit source dispatch in `admin/sync/route.ts`.
