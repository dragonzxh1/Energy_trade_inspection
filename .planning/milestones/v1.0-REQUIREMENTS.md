# Requirements: Energy Trade Inspection (ETI)

**Defined:** 2026-04-13
**Core Value:** Give energy traders instant, defensible answers on whether a counterparty is safe to trade with

## v1 Requirements

This milestone upgrades ETI from a data aggregator (~55% complete) to a trade fraud decision engine (~85% complete) by closing data source gaps, sharpening the verdict engine, and hardening the architecture.

### Data Sources

- [x] **DATASRC-01**: System syncs FCA (UK) warning list and displays badge on entity pages when warning present
- [x] **DATASRC-02**: System syncs FINMA (Switzerland) and SFC (Hong Kong) warning lists and displays badges on entity pages
- [x] **DATASRC-03**: System syncs MAS (Singapore) investor alert list and displays badge on entity pages when warning present
- [x] **DATASRC-04**: System syncs DFSA (Dubai DIFC), SCA (UAE federal), and CMA Oman warning lists and displays badges on entity pages
- [x] **DATASRC-05**: System checks domain WHOIS via RDAP and exposes registration age, registrar, and privacy shield status
- [x] **DATASRC-06**: System checks email domain MX records and SPF/DKIM/DMARC configuration to detect disposable/fraudulent domains

### Decision Engine

- [ ] **DECISION-01**: Risk labels distinguish between `sanctioned` (OFAC/EU/UN), `warning_listed` (FCA/MAS/DFSA/SCA/CMA/FINMA/SFC), and `export_restricted` (BIS/ECFR) — each displays with distinct badge color and tooltip naming the source
- [ ] **DECISION-02**: Trade risk check returns structured Safe / Review / Block verdict (not just risk flags) with explicit reason codes per flag
- [ ] **DECISION-03**: Each reason code in the trade verdict maps to a human-readable explanation and the data source that triggered it
- [ ] **DECISION-04**: Compliance officer can export a trade verdict as a PDF audit trail showing verdict + reason codes + data sources + related-party flags + UTC timestamp
- [ ] **DECISION-05**: If any direct director or shareholder of the counterparty is sanctioned or warning-listed, a `related_party_risk` flag is raised in the verdict with the name and list source (1-hop only, using existing director data from Companies House/ACRA/Zefix)

### Scoring Engine

- [ ] **SCORE-01**: Trading Track Record dimension (max 25 pts) is calculated from verifiable trade data — score is non-zero for entities with documentable trade history
- [ ] **SCORE-02**: Behavioral pattern scoring detects shell company signals: anonymous registration, zero employee count, no verifiable web presence, newly registered (< 6 months)
- [ ] **SCORE-03**: Paid users can view per-dimension score breakdown (Entity Existence, Asset Reality, Document Consistency, Community Reputation, Trading Track Record) with contributing factors

### Architecture & Security

- [ ] **ARCH-01**: Centralized `middleware.ts` enforces authentication on all protected routes — per-route `auth()` checks are removed from existing API routes
- [ ] **ARCH-02**: OpenSanctions API has circuit breaker — when API is unavailable, screening returns `status: degraded` with cached data and does not fail silently
- [ ] **ARCH-03**: Admin sync endpoint `/api/admin/sync` verifies admin role before executing — admin authorization is explicit and audited
- [ ] **ARCH-04**: Python intelligence wrapper (`src/lib/server/intelligence.ts`) resolves Python binary path correctly on both Windows (`.venv/Scripts/python.exe`) and Linux (`.venv/bin/python`) — Tavily queries succeed on Linux production

### Admin Operations

- [ ] **ADMIN-01**: Admin dashboard displays data sync job history — each run shows source, status (success/failure), record count, duration, and error message if failed
- [ ] **ADMIN-02**: Admin dashboard displays full user list with email, plan, registration date, last active date, and quota consumed — searchable and sortable
- [ ] **ADMIN-03**: Admin can manually change any user's plan (free / starter / enterprise) from the dashboard — change takes effect immediately without Stripe webhook
- [ ] **ADMIN-04**: Admin dashboard shows platform usage statistics: total users, plan distribution, daily new registrations for past 30 days, and top entity types screened

## Out of Scope

| Feature | Reason |
|---------|--------|
| Full KYC/AML platform | Different product category — ongoing monitoring SLAs require compliance infrastructure we don't have |
| Real-time adverse media | Requires content pipeline and NLP — high cost, future milestone |
| SWIFT / payment network checks | Requires financial institution partnerships — out of reach |
| Vessel physical inspection scheduling | Logistics integration — unrelated to compliance screening |
| ORM (Prisma/Drizzle) migration | Established raw SQL pattern; migration risk > benefit |
| Automated test suite | Not in scope for this milestone — explicit technical debt |
| Mobile app | Web-first product; compliance tools are desktop-native |
